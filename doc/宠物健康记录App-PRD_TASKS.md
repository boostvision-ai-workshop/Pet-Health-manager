# 宠伴健康（宠物健康记录 App）- 分步骤可执行任务拆解

本文档将 [宠物健康记录App-PRD-MVP.md](./宠物健康记录App-PRD-MVP.md) 拆解为可编译、可验收的步骤。技术选型见 PRD §11（Flutter Android、Riverpod 或 Provider、go_router、Hive 或 SharedPreferences、fl_chart、flutter_local_notifications）。

## 步骤 1：工程骨架与依赖

**目标**：建立可运行的 Flutter Android 工程，并拉齐 PRD 建议的技术栈依赖与目录习惯。

**范围**：

- 创建或确认工程为 **仅 Android** 首发（竖屏）。
- `pubspec.yaml`：`go_router`、`flutter_riverpod`（或 Provider）、`hive` + `hive_flutter`（或 `shared_preferences`）、`fl_chart`、`flutter_local_notifications`、`intl`（日期格式化）等。
- `lib/` 建议结构：`app/`、`models/`、`services/`、`features/home|timeline|profile|pet_profile|events|reminders/`、`shared/`（对齐 PRD §11.2）。

**验收**：

- **编译**：`flutter pub get` 成功；`flutter analyze` 无 error；`flutter build apk`（或 `flutter run` 在模拟器）通过。
- **可测**：无需业务测试，以编译与工程可启动为准。

**依赖**：无

---

## 步骤 2：领域模型与枚举

**目标**：用类型表达 PRD §9 建议的本地数据形状，并把字段规则编码为可复用校验（可在 UI 或服务层调用）。

**范围**：

- `Pet`：`id`、`name`、`birthday`、`gender`、`breed`、`adoptionDate`、`avatar`、`latestWeight`（或运行时由体重事件推导，二选一并文档说明）。`gender` 持久化可为 `male`/`female`/`unknown`，**用户可见文案**为 弟弟、妹妹、未知（见 PRD §7.5，非「公」「母」）。
- 事件：`type`（`weight` | `deworm` | `spayNeuter` | `vaccine` | `physicalNote`）、`dateTime`、`value`/`unit`（体重 kg）、`note`，`id`。
- `Reminder`：`eventId`、`title`、`dueDate`、`status`（`todo` | `done`）。
- 校验：体重事件必填数值；备注最大 200 字；档案姓名必填、日期不晚于今天（PRD §7.4、§7.5）。

**验收**：

- **编译**：同上。
- **可测**：**单元测试**：模型序列化/等价性；校验函数对合法/非法输入的断言（如未来生日、超长备注）。

**依赖**：步骤 1

---

## 步骤 3：本地持久化仓储

**目标**：所有数据仅本地持久化（PRD §8.6），对外提供宠物、事件、提醒的稳定 CRUD 接口。

**范围**：

- 单宠物：仅存一条档案或用一个 `pet_001` 约定；不支持第二只宠物（PRD §8.1）。
- 事件：增删改查；删除后大事记移除（PRD §8.3）。
- 提醒：与 `eventId` 关联；完成态可更新。
- 若用 Hive：定义 `TypeAdapter` 或使用 JSON 字符串存储策略（团队选一种并统一）。

**验收**：

- **编译**：通过。
- **可测**：**单元测试**：写入宠物与事件后读取一致；删除事件后列表不含该条；更新提醒状态后持久化可读。

**依赖**：步骤 2

---

## 步骤 4：里程碑与健康状态服务

**目标**：实现 PRD §8.4、§8.7 的可测试纯逻辑，避免散落在 Widget 内。

**范围**：

- 里程碑：到家天数；距上次驱虫/疫苗；绝育后天数（无最近一次事件则「待补充」）。
- 健康状态：稳定 / 待处理 / 需关注 的判定（未来 7 天提醒、逾期、30 天内有记录、60 天无记录等条件，严格按 PRD §8.7）。

**验收**：

- **编译**：通过。
- **可测**：**单元测试**：给定固定「今天」、事件与提醒列表，断言状态与各项展示文案（含「待补充」）。

**依赖**：步骤 2、3

---

## 步骤 5：体重趋势数据准备

**目标**：为首页体重卡提供统一数据源：最近 30 天、排序、点数判断（PRD §8.2）。

**范围**：

- 从仓储筛选 `type == weight` 的记录，映射为 `(dateTime, value)` 序列。
- 分支：无数据 / 一条 / 两条及以上，供 UI 选择空态、单点、折线图。

**验收**：

- **编译**：通过。
- **可测**：**单元测试**：Mock 仓储或注入列表，断言 30 天窗口与分支条件。

**依赖**：步骤 2、3

---

## 步骤 6：路由与三 Tab 壳

**目标**：落地信息架构三条主路径：首页、大事记、我的（PRD §6），并保证导航可达。

**范围**：

- `go_router`（或项目选定方案）配置底部 `NavigationBar`/`BottomNavigationBar` 三 Tab。
- 子路由预留：创建/编辑档案、新增事件（可先占位 `Scaffold` + 标题）。

**验收**：

- **编译**：`flutter build apk` 或通过 `flutter run`。
- **人工**：冷启动 → 在三 Tab 间切换，占位页标题正确、无路由错误。

**依赖**：步骤 1

---

## 步骤 7：主题与设计 Token

**目标**：把 PRD §10.2 的颜色、圆角、字号档位落到 `ThemeData`，减少各页硬编码。

**范围**：

- 颜色：主色 `#5F7A74`、背景 `#F5F5F7`、卡片白、一级/二级文字、分割线。
- 圆角：主卡约 22–24、次级 16–18、按钮 14–16。
- 字体：优先平台字体方案；中文回退说明可在 `README` 或代码注释一行（不单写新 markdown 文档除非团队要求）。

**验收**：

- **编译**：通过。
- **人工**：任一 Tab 占位页使用主题色与卡片样式，观感符合「简约卡片化」。

**依赖**：步骤 6

---

## 步骤 8：创建/编辑宠物档案页

**目标**：完成 PRD §7.5 的表单与持久化，作为单宠物唯一档案入口。

**范围**：

- 字段：头像、姓名、出生、性别、品种、到家日期、初始体重。其中性别为 弟弟、妹妹、未知 三选一（与 PRD §7.5 一致）。
- 校验与按钮：姓名必填；日期不晚于今天；保存写入仓储。
- 支持「编辑」回显（与步骤 14 联动时可把本步仅做到保存与读取一条档案）。

**验收**：

- **编译**：通过。
- **人工**：填写保存 → 重启 App 档案仍在；非法日期有提示无法保存。

**依赖**：步骤 3、6、7

---

## 步骤 9：首次无档案引导

**目标**：实现 PRD §8.1：无档案时直接进入创建档案流程，有档案后进入主 Tab。

**范围**：

- 启动时读仓储：无 `Pet` → 路由到创建档案；有则进 `Shell` 三 Tab。
- 保存成功后进入主界面。

**验收**：

- **编译**：通过。
- **人工**：清除本地数据后首次启动落到创建页；保存后进入 Tab；二次启动进入首页。

**依赖**：步骤 6、8

---

## 步骤 10：新增健康事件页

**目标**：统一表单记录五类事件（PRD §7.4），并写入仓储；为提醒子对象打基础。

**范围**：

- 类型选择：体重、驱虫、绝育、疫苗、体检笔记。
- 体重类型显示数值输入，单位固定 kg。
- **下次提醒**（可空，存 `Reminder`）：`dueDate` 为**所选日历日 09:00**；**快速选日期** `ActionChip`：明天、7/30/90 天后；**选择具体日期** `showDatePicker` + 行内清除。
- 保存后 `latestWeight` 若需更新，在本步或仓储层统一更新策略。

**验收**：

- **编译**：通过。
- **人工**：各保存一条事件 → 重启后仍存在；体重无数值时不能提交；备注超长拦截；**快速选日与日历**设定后展示与落库为当日 9:00；清除后无提醒。

**依赖**：步骤 3、6、8（档案应先存在）

---

## 步骤 11：首页主卡与里程碑卡

**目标**：首页「第一眼」信息层（PRD §7.1）：摘要卡 + 里程碑卡 + 健康状态。

**范围**：

- 摘要卡：头像、姓名、出生、性别（弟弟/妹妹/未知）、品种、最新体重、健康状态（调用步骤 4）。
- 里程碑卡：到家天数、距上次驱虫、距上次疫苗、绝育后天数或「待补充」。
- 新增健康事件**不**在首页放全局 `floatingActionButton`；主入口在「大事记」页右下角（见步骤 13）。

**验收**：

- **编译**：通过。
- **人工**：与仓储数据一致；缺乏历史事件时显示「待补充」；健康状态与提醒/记录联动合理。

**依赖**：步骤 4、7、9、10

---

## 步骤 12：体重趋势卡

**目标**：实现 PRD §8.2 与首页体重区块：30 天默认、空态、单点、折线。

**范围**：

- 使用步骤 5 的数据；`fl_chart` 折线图（≥2 点）。
- 无数据：引导「记录第一条体重」并可跳转新增页。
- 一条：单点展示（非折线）。

**验收**：

- **编译**：通过。
- **人工**：0/1/多 条体重下三种 UI 正确；日期范围符合「最近 30 天」语义（与 PRD一致的截断即可）。

**依赖**：步骤 5、10、11

---

## 步骤 13：大事记列表与筛选

**目标**：大事记时间线（PRD §7.2、§8.3）：倒序、筛选、绝育默认排除。

**范围**：

- 列表项：类型、时间、备注或数值、是否关联提醒（可从 `Reminder` 反查）。
- 筛选 Chips：全部 / 体重 / 健康事件（健康事件含驱虫、疫苗、体检笔记等，按 PRD 大类）。
- **绝育记录写入仓储但不出现在默认时间线**（PRD §7.2、§8.3）；删除事件后列表更新。
- 右下角 **「记录」** 扩展悬浮按钮：跳转统一新增健康事件页（`/events/add`）；空状态文案需引导通过该入口添加。

**验收**：

- **编译**：通过。
- **人工**：混有多类型记录时排序正确；筛选「体重」仅体重；绝育有存档但列表默认不出现；删除一条后消失；右下角「记录」可进入新增页且与 PRD 框线一致。

**依赖**：步骤 3、10、6、7

---

## 步骤 14：我的页与编辑入口

**目标**：PRD §7.3：完整档案展示、关键指标、编辑宠物档案入口。

**范围**：

- 档案卡 + 关键指标（当前体重、到家天数等，与 PRD 框线一致）。
- 「编辑宠物档案」进入步骤 8 表单；「本地提醒设置」可先路由到占位再步骤 16 填满。

**验收**：

- **编译**：通过。
- **人工**：字段与首页/仓储一致；编辑保存后返回各页数据更新。

**依赖**：步骤 7、8、11、12

---

## 步骤 15：提醒数据与首页近期提醒

**目标**：提醒轻量模型与首页「最近 3 条」、完成操作（PRD §7.1、§8.5）。

**范围**：

- 保存事件时若填写「下次提醒日期」，创建/更新 `Reminder`（`todo`）。
- 首页展示最近 3 条（含逾期文案）；标记 `done` 后首页与大事记「是否关联提醒」一致。

**验收**：

- **编译**：通过。
- **人工**：新增带提醒的事件 → 首页出现；标记完成 → 状态变更且列表刷新。

**依赖**：步骤 3、10、11

---

## 步骤 16：应用内提醒与系统通知（可分期）

**目标**：完成 PRD §8.5：至少 App 内提醒可靠；Android 优先系统通知。

**范围**：

- **P1 底线**：应用内列表（「我的」或独立入口）展示待处理/已完成；到期前展示逻辑与首页一致。
- **增强**：`flutter_local_notifications` 预约/取消与事件同步；权限处理（Android 13+ 通知权限）。

**验收**：

- **编译**：通过。
- **人工**：设置提醒 → App 内可见；若已接系统通知 → 到达时间可触发（真机或模拟器以环境为准）；完成提醒后通知策略不重复骚扰（实现组自行定义合理行为）。

**依赖**：步骤 15

---

## 步骤 17：空状态与边界体验

**目标**：覆盖 PRD §10.8 与验收体验：无体重、无大事记、筛选无结果、必填未完成、保存失败提示等。

**范围**：

- 各页 `EmptyState` 组件；表单校验错误提示；可选轻量加载/错误态。
- P2：空状态美化、通知细节可归本步或独立小迭代（与 PRD §13 对齐）。

**验收**：

- **编译**：通过。
- **人工**：按 PRD §12 清单走查：建档、记事件、大事记筛选、里程碑、提醒、编辑档案均在边界下可用且不崩溃。

**依赖**：步骤 9–16（按实际完成度抽查）
